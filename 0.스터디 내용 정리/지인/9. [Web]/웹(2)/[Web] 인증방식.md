# 인증 방식

## HTTP 인증 방식 개요

웹 서비스는 클라이언트와 서버가 HTTP 프로토콜로 통신하며, 각 요청과 응답은 독립적으로 이루어집니다.

하지만 HTTP는 기본적으로 무상태(Stateless) 특징을 지녀, 로그인 상태 등 사용자의 맥락을 서버가 기억할 수 없습니다.

이에 따라 웹에서는 클라이언트의 신원을 증명하고, 보호된 자원에 대한 적절한 권한을 부여하기 위해 다양한 인증 방식이 필요합니다.

<br/>

## 대표적인 인증 방식

### Basic 인증 (Basic Authentication)

가장 단순한 인증 방식으로, 사용자의 아이디(ID)와 비밀번호(PW)를 ":"로 연결해 base64로 인코딩한 문자열을 'Basic'과 함께 Authorization 헤더에 담아 전송합니다.

```
Authorization: Basic base64({USERNAME}:{PASSWORD})
```

- 장점
  구현이 간단하여 빠르게 적용할 수 있다.

- 단점
  base64는 암호화가 아니므로, 네트워크 상에서 쉽게 정보가 노출될 위험이 있다.
  반드시 HTTPS(SSL/TLS)로 보호해야 하며, 서버에 사용자 목록을 저장할 필요가 있고, 세밀한 권한 제어나 대규모 스케일에 한계가 있다.

<br/>

### Bearer 인증 (Bearer Authentication)

Bearer 인증은 OAuth 2.0 등에서 사용하는 토큰 기반 인증 방식입니다.

사용자는 권한이 부여된 토큰(Access Token)을 발급받아, 이후 모든 요청의 Authorization 헤더에 'Bearer' 키워드와 함께 전송합니다.

```
Authorization: Bearer {access_token}
```

- 장점

  - 요청마다 아이디/비밀번호를 반복 입력할 필요 없이 토큰만 전송한다.

  - 서버는 토큰의 유효성과 권한만 검증하면 되므로 빠르고 확장성 있는 시스템 구현이 가능하다.

  - OAuth, JWT 등과 연계하여 소셜 로그인 등 다양한 인증·인가 시나리오에 적용할 수 있다.

  - 토큰의 유효기간이나 권한을 미세하게 관리·제어할 수 있다.

- 단점

  - 토큰이 유출될 경우, 해당 토큰으로 누구나 자원에 접근할 수 있으므로 보안 관리가 매우 중요하다.

  - 초기 인증/권한 서버 구성 등이 Basic 인증에 비해 복잡할 수 있다.

<br/>

### 세션(Session) 기반 인증

서버가 인증된 사용자 정보 및 세션 ID를 자체적으로 저장하고, 클라이언트에는 세션 ID만 쿠키 등으로 전달하는 방식입니다.

서버에서 상태(세션)를 관리하므로 보안에 강점이 있지만, 서버의 부하가 늘고 분산 시스템 확장 시 복잡성이 커집니다.

<br/>

### 토큰(Token) 기반 인증 (JWT 등)

서버가 인증에 성공한 사용자에게 자체적으로 정보가 포함된 토큰(JWT 등)을 발급합니다.

클라이언트는 이를 저장해두었다가 요청마다 직접 HTTP 헤더에 포함시켜 서버로 전송합니다.

서버는 별도의 저장소 없이 토큰 유효성만 검증하면 되므로 무상태(Stateless) 구조에 적합합니다.

<br/>

### 보다 안전한 인증 방식으로의 진화

실제 서비스 환경에서는 단순 Basic 인증보다 Bearer, JWT, OAuth 등 토큰 기반 인증 방식이 널리 사용됩니다.

이외에도 2단계 인증(2FA), SMS·OTP 인증, SAML 등 다양한 보안 강화 방법이 결합되어 쓰이기도 합니다.

각 인증 방식은 사용 환경, 보안 수준, 시스템 확장성, 사용자 경험 등을 고려해 선택하면 됩니다.

클라이언트와 서버 구조 및 보안 요구사항에 따라 적합한 인증 방식을 도입하는 것이 중요합니다.
